# 📖 01 — Prompt 模板 Cookbook

> 🎯 **目标**：建立自己的 Prompt 模板武器库，任何任务都能快速拿出高质量模板，不再从零写起。
> ⏱️ 预计时间：2 天
> 📦 收录：**6 大类 26 个模板**

---

## 📋 使用说明

### 模板格式约定

- `{{变量}}` = 你需要替换的内容
- `[可选]` = 可选配置项
- 每个模板末尾标注了适用模型和推荐参数

### 模板分类速查

| 类别 | 模板数 | 主要使用场景 |
|------|--------|-------------|
| 📝 文本处理 | 6 | 翻译/摘要/改写/邮件/纪要/文案 |
| 💻 代码生成 | 5 | 函数/SQL/Shell/测试/Bug修复 |
| 🔍 分析推理 | 4 | 代码审查/方案分析/日志诊断/数据洞察 |
| 📚 知识问答 | 3 | RAG问答/概念解释/FAQ生成 |
| 📐 结构化输出 | 3 | JSON约束/表格生成/分类标注 |
| 🎭 角色扮演 | 4 | 面试官/代码导师/文档写手/产品经理 |

## 📝 A. 文本处理类

### A1. 翻译助手 🌐

**适用场景**：翻译技术文档、README、论文摘要，需保持格式和专业术语

````markdown
将以下 {{源语言}} 文本翻译成 {{目标语言}}。

**要求：**
- 保持原文的 Markdown 格式（标题、列表、代码块、链接）
- 代码块和 URL 不翻译
- 技术术语使用行业标准译法，首次出现标注英文原文
- 专业名词保留英文（如 API、Transformer、backward 等）

**原文：**
{{待翻译文本}}
````

**优化技巧**：翻译论文时加一句「对不确定的术语，在译文后用 [原文: xxx] 标注」。

### A2. 文本摘要 📄

**适用场景**：快速理解长文档、生成周报摘要

````markdown
请对以下文本生成摘要。

**模式**：{{简短摘要|详细摘要|要点式}}

[如果是要点式摘要]
以 bullet points 形式列出 3-7 个要点，每个要点不超过 30 字。

[如果是简短摘要]
用 2-3 句话概括核心内容，总字数不超过 100 字。

[如果是详细摘要]
保留原文的论证逻辑，详细复述主要观点、关键证据和结论，字数不超过原文的 30%。

**文本：**
{{待摘要文本}}
````

**期望输出示例**（要点式）：
```
- Transformer 架构于 2017 年提出，用 Self-Attention 替代 RNN 的循环结构
- 核心创新：并行计算能力 + 长程依赖捕获
- 已成为 GPT、BERT、Llama 等主流模型的基础架构
```

### A3. 文本改写 ✏️

**适用场景**：论文润色、技术博客通俗化、正式邮件撰写

````markdown
- 「缩短原文」：保留核心信息，压缩到原文的 {{50%|30%}}，删掉修饰和重复

**原文：**
{{待改写文本}}
````

**期望输出示例**（学术→通俗）：
> 原文："The Transformer employs a multi-head self-attention mechanism to capture long-range dependencies."
> 改写："Transformer 就像一个超级读者，能同时关注一句话中的所有词，而不是一个词一个词地读。"

## 📐 E. 结构化输出类

### E1. JSON Schema 约束输出 📋

````markdown
请分析以下文本，并按指定 JSON Schema 输出。

**文本**：
{{待分析文本}}

**JSON Schema**（你的输出必须符合此结构）：
```json
{
  "type": "object",
  "properties": {
    "sentiment": {
      "type": "string",
      "enum": ["positive", "negative", "neutral"]
    },
    "confidence": {
      "type": "number",
      "minimum": 0,
      "maximum": 1
    },
    "keywords": {
      "type": "array",
      "items": {"type": "string"},
      "maxItems": 5
    },
    "summary": {
      "type": "string",
      "maxLength": 100
    }
  },
  "required": ["sentiment", "confidence", "keywords", "summary"]
}
```

**格式要求**：只输出 JSON，不要包含任何 Markdown 代码块标记或额外解释。
````

**推荐参数**：`temperature=0`（要求确定性输出时）

## 🚨 Prompt 常见翻车现场

| 现象 | 原因 | 解决 |
|------|------|------|
| 模型输出不是 JSON | 没加「只输出 JSON，不要其他文字」 | 加明确约束 + `temperature=0` |
| 翻译丢失了 Markdown 格式 | 没告诉模型保留格式 | 第一句就写「保持 Markdown 格式」 |
| 摘要太短/太长 | 没指定字数和模式 | 三种模式明确选择 |
| 代码有 bug 但能跑 | 边界情况没覆盖 | 模板中列出边界要求 |
| Few-Shot 示例起了反作用 | 示例有错误或格式不一致 | 每个示例都经过测试验证 |
| 模型开始自由发挥 | 没设定角色边界 | 加「你不能...」约束句 |

## 🔧 模板使用的最佳实践

### 如何选择 Temperature

| 任务类型 | Temperature | 原因 |
|----------|------------|------|
| JSON 结构化输出 | 0 | 要求确定性 |
| SQL/代码生成 | 0-0.2 | 代码需要精确 |
| 翻译 | 0.1-0.3 | 保留原文信息 |
| 摘要 | 0.3-0.5 | 可以有表达灵活性 |
| 文案/写作 | 0.7-0.9 | 需要创意 |

### 模板调试四步法

```
1. 先用最简单的 Prompt 跑一遍（Zero-Shot）
2. 如果效果不好，加一个示例（One-Shot）
3. 如果还不好，加 2-3 个示例 + 格式约束（Few-Shot）
4. 如果特定场景反复失败，拆成多步（Chain）
```
